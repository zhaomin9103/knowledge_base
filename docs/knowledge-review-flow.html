关键规则说明（v3.0 核心变更）

  • 初审人拥有终点决策权：初审通过时可选择「通过并生效」（直接生成版本，跳过复审）或「提交复审」（转入复审环节），决策理由可选填。
  • 四个独立角色：
    • 创建者：拥有初审权 + 成员管理权（独立角色），提交需复审后生效
    • 复审人：拥有复审权（最高审核级），提交直接生效
    • 初审人：拥有初审权（含终点决策权），提交需复审后生效
    • 维护人员：只能提交，需经初审（初审人决定是否需复审）
  • 维护人员的任何变更必须经初审，初审人决定是否需要复审。
  • 初审人 / 创建者自己的提交跳过初审，直接进入待复审环节。
  • 复审人的提交无需审核，直接生效并生成版本。
  • 流程降级规则（角色配置不完整时）：
    • 没有复审人（secondReviewerIds.length === 0）：
      • 维护人员提交 → 初审人审核 → 只能选择「通过并生效」（自动隐藏「提交复审」按钮）
      • 初审人/创建者提交 → 直接生效（自动跳过复审环节）
      • UI 提示：提交确认弹窗显示"您的提交将直接生效（当前知识库未配置复审人）"
    • 只有创建者（极端情况）：
      • 创建者提交 → 直接生效（无需审核）
      • 维护人员提交 → 创建者初审 → 直接生效
  • 文档审核中锁定：文档存在 pending_first_review 或 pending_second_review 状态的记录时，禁止其他增删改操作。一个文档同时只能有一个 pending 记录。
  • 驳回后可立即重新提交：rejected 状态不占用 pending 约束，驳回后可立即修改并重新提交。
  • 乐观锁并发控制：每条 ReviewRequest 包含 version 字段，审核操作提交时校验版本号，防止多人同时审核同一记录。
  • 版本生成规则：
    • 初审直接通过：生成版本，仅记录初审人（approvedBy = "first"）
    • 复审通过：生成版本，记录初审人+复审人（approvedBy = "second"）
    • 复审人直接提交：生成版本，无审核人记录（approvedBy = "direct"）
    • 降级场景：生成版本，标记降级原因
  • 可追溯性：所有审核操作写入「审核记录」，所有生效变更写入「操作记录」，支持完整的审计链路。
